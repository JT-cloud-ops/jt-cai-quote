# 估價單專案規劃書

## 0. 評估範圍與結論

本次評估範圍為 `quotation-app`，技術棧為 Vite + React + TypeScript。主功能包含單張類、冊子類、百貨類報價輸入、即時預覽、列印、JSON 匯入/匯出、localStorage 歷史紀錄與 LINE LIFF 分享。

整體結論：目前專案適合先定位為「內部前端工具」。若要擴大到多人共用、正式客戶分享或長期保存，第一優先不是加功能，而是先補上資料驗證、隱私保護、建置穩定性與元件拆分。

協作註記：本環境沒有可直接指定 Gemini 的工具入口；已改用可用的本地子代理做 read-only 掃描，並已將其回覆整合進本文件。以下結論以實際程式碼掃描、本地 build 嘗試與子代理補充結果為準。

- `src/components/QuotationForm.tsx`：集中處理表單、歷史紀錄、客戶資料、JSON 匯入/匯出、LIFF 分享、列印。

## 2. 安全性評估

### 高優先級

1. 匯入資料缺少 schema 驗證

   `src/App.tsx` 會從 URL `?import=` 讀取 base64 JSON 並直接 `JSON.parse` 後寫入狀態；`src/components/QuotationForm.tsx` 也會直接讀取使用者匯入的 JSON 檔。雖然 React 預設會 escape 文字，XSS 風險相對可控，但資料結構污染、超大 payload、錯誤型別、非法金額與破壞 UI 的內容仍是實際風險。

   建議：新增 `src/schemas/quotationSchema.ts`，使用 Zod 或自寫 type guard 驗證 `quotationType`、items、bookletJobs、數字欄位、字串長度與陣列上限。

2. 客戶與業務個資存在 localStorage

   `quotationHistory`、`customerDatabase`、`lastSalesName`、`lastSalesMobile` 都存在瀏覽器 localStorage。localStorage 沒有加密，任何同源腳本都可讀取，電腦共用時也容易外洩。

   建議：短期加入「清除本機資料」與「隱私提示」；中期改為 IndexedDB 並提供匯出/備份；若未來上雲，客戶資料必須進後端權限控管。

3. LINE 分享連結包含完整報價資料

   目前 LIFF 分享 URL 把完整報價資料 base64 後放在 query string。base64 不是加密，任何取得連結的人都能還原客戶、品項、金額與聯絡資料。

   建議：短期在分享前提示「此連結含完整報價資料」；中期改成只分享摘要或一次性 token；正式版應由後端保存報價內容，前端分享不可逆 token。

4. 開發伺服器固定開放區網

   `vite.config.ts` 固定 `server.host: '0.0.0.0'`，且 `啟動估價單系統.bat` 又使用 `--host`。同區網其他裝置可能連入開發伺服器。若內含客戶資料或瀏覽器資料被操作，風險會提高。

   建議：日常使用改為只綁 `localhost`；需要手機/LINE 測試時才使用另一個明確命名的 `啟動估價單系統-區網測試.bat`。

5. 外部 LIFF SDK 缺少瀏覽器安全邊界設定

   `index.html` 載入 `https://static.line-scdn.net/liff/edge/2/sdk.js`。這在 LIFF 情境合理，但目前沒有 Content Security Policy。建議正式部署時加入 CSP，限制 `script-src`、`connect-src`、`frame-ancestors` 只允許 LINE 與自家網域。

### 中優先級

- `Math.random()` 用於產生 UI id，安全性用途不足。若只是 React key 影響不大，但建議改成 `crypto.randomUUID()`。
- LIFF ID 寫死在前端，這本身不是 secret，但應集中於 `src/config/env.ts` 或 `.env`，方便區分測試/正式環境。
- JSON 匯出檔名來自 prompt，應移除 Windows 不合法字元並限制長度。
- 金額與日期目前多為字串，容易出現負數、非數字、極大數、空白與格式不一致。
- `index.html` 載入外部 LIFF SDK，需固定來源、記錄版本策略，並準備 SDK 載入失敗時的降級流程。

### 低優先級

- 目前未看到 `dangerouslySetInnerHTML`，React 預設渲染可降低直接 XSS 風險。
- 目前是純前端工具，沒有伺服器端 API、資料庫密碼或明顯後端注入面。

## 3. 可擴充功能建議

### 第一階段：讓現有工具穩定可用

- 報價單編號：依公司別、民國年月日、流水號產生，例如 `JC-1150529-001`。
- 修訂版管理：同一報價單可有 `v1/v2/v3`，保留修改紀錄。
- 公司資料設定檔：把公司抬頭、地址、電話、統編、條款從 `QuotationPreview.tsx` 移到設定檔。
- 常用客戶管理：搜尋、編輯、刪除、匯入/匯出客戶資料。
- 常用品項模板：紙張、尺寸、加工、裝訂、單位、稅別可快速選取。
- 金額驗證與格式化：輸入時限制數字，顯示千分位，避免空白或非數字進入計算。

### 第二階段：提升報價效率

- 報價模板：單張、冊子、百貨各自可套用預設欄位。
- 快速複製品項/整張報價單。
- 紙張與加工價格表：建立紙張單價、加工費、裝訂費基礎資料。
- 自動計價規則：用規則計算基本小計，再允許業務手動覆寫。
- 條款模板：付款方式、交期、有效期限、注意事項可套用。
- PDF 檔名規則：公司別、客戶、品名、報價日期自動產生。

### 第三階段：多人與正式交付

- 後端儲存：報價單、客戶、商品、價格表集中管理。
- 權限角色：業務、主管、管理員、唯讀。
- 核准流程：超過金額或毛利門檻需主管核准。
- 報表：月報價數、成交率、客戶排行、品項排行。
- 客戶線上確認：客戶可透過安全連結確認報價。
- Excel/PDF 正式匯出：對齊現有 `捷采印刷報價單(空白).xls` 樣式。

### 拆分順序

1. 先抽出 domain 純函式

   把 `createEmptyItem`、`createEmptyBookletJob`、金額計算、稅額計算、日期格式化搬出元件。這一步風險低，最能降低重複邏輯。

2. 再抽出 storage

   把 localStorage key、讀寫、JSON parse fallback 集中管理。之後若改 IndexedDB 或後端 API，畫面元件不用大改。

3. 再抽出 hooks 與 integrations/line

   把 LIFF init、login、shareTargetPicker、分享文字產生移出 `QuotationForm.tsx`。建議拆為 `useLineShare`、`lineShareMessage.ts`、`liffClient.ts`。畫面只負責按鈕與錯誤提示。

   `QuotationPreview.tsx` 目前同時包含公司資料、表格、金額、條款與 footer。先把公司設定抽到 `companies.ts`，再拆成 `CompanyHeader`、`QuotationTable`、`TotalsRows`、`ContractFooter`。

## 5. 建議資料模型調整

目前金額、數量、日期多為 string，畫面輸入方便，但計算與驗證成本高。建議分成「表單草稿」與「已驗證報價」兩層。

```ts
type QuotationDraft = {
  quotationType: 'single' | 'booklet' | 'dept';
  customerName: string;
  items: QuotationItemDraft[];
};

type ValidQuotation = {
  quotationType: 'single' | 'booklet' | 'dept';
  customerName: string;
  items: QuotationItem[];
  totals: QuotationTotals;
};
```

原則：

- 表單層保留 string。
- 儲存/分享/列印前轉成 validated model。
- 所有計算只接受 validated model。
- 匯入 JSON 一律先通過 schema，再進入 UI。

## 7. 下一步建議

最務實的下一步是先做 Milestone 1。原因是目前所有後續功能都依賴資料格式可信、建置流程穩定、匯入匯出安全。如果先加功能，`QuotationForm.tsx` 與 `QuotationPreview.tsx` 會繼續膨脹，後續重構成本會明顯上升。

建議第一個 PR/修改批次只做：

- 新增 `src/domain/quotationFactory.ts`
- 新增 `src/domain/quotationCalculations.ts`
- 新增 `src/storage/localStorageRepository.ts`
- 新增 `src/domain/quotationValidation.ts`
- 調整 `App.tsx` 與 `QuotationForm.tsx` 使用上述模組

完成後再拆 Preview 與 LIFF，風險最低。
